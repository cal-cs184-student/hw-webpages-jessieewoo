Task 1: Drawing Single-Color Triangles

Walk Through How You Rasterize Triangles in Your Own Words

  • For Task 1, I rasterize a single-color triangle by first defining a helper function called triangle_test, which performs a 2D cross product sign test. This function takes an edge of the triangle and a sample point, and computes whether the point lies to the left or right of that edge.
  • To check if a point is inside the triangle, I evaluate this cross product test for all three edges. For each edge, I compute the signed area value using the cross product formula. This gives me a positive or negative result depending on which side of the edge the point lies on.
  • If the signs of all three edge tests are consistent (either all non-negative or all non-positive), then the point lies inside the triangle. If the signs are mixed, that means the point lies outside the triangle. This works because a point inside a triangle must lie on the same side of all three directed edges.
  • Next, I compute the triangle’s bounding box. The bounding box is the smallest axis-aligned rectangle that completely contains the triangle. I find this by taking the minimum and maximum x and y coordinates of the three triangle vertices.
  • Since the vertex coordinates are floating-point values, I convert the bounding box limits into integer pixel indices using floor and ceil. This allows me to safely iterate over discrete pixels while covering the entire triangle area.
  • I then iterate over every pixel inside the bounding box. For each pixel, I compute the sample location at the pixel center, such as (x + 0.5, y + 0.5), because we are using one sample per pixel.
  • For each sample point, I run the triangle_test on all three edges. If the sample point passes all three edge tests and the signs are consistent, it is inside the triangle. If it is inside, I fill that pixel with the triangle’s color.

Runtime Analysis

My implementation iterates only over the triangle’s bounding box and performs a constant amount of work per sample (three edge tests and a few comparisons). Therefore, the runtime is proportional to the number of samples in the bounding box, O(B), which is asymptotically no worse than an algorithm that checks each sample once within the bounding box.

Task 2: Antialiasing by Supersampling

Walkthrough of Algorithm and Data Structures

  • To implement supersampling, I changed how each pixel stores its color. Instead of storing just one color per pixel, I use a sample_buffer that stores multiple color values per pixel. The number of stored colors per pixel equals sample_rate. All of a pixel’s subsamples are stored next to each other in memory.
  • For each triangle, I first compute its bounding box by taking the minimum and maximum x and y values of the three vertices. I clamp those bounds so I never go outside the screen dimensions.
  • Then I compute N as the square root of sample_rate. This means each pixel is divided into an N by N grid of evenly spaced subsamples. For example, if the sample rate is 4, each pixel becomes a 2 by 2 grid. If the sample rate is 16, each pixel becomes a 4 by 4 grid.
  • For every pixel inside the bounding box, I loop over all N by N subsamples. For each subsample, I compute its exact position inside the pixel. I place the subsample at the center of each small grid cell by offsetting from the pixel’s top-left corner using (i + 0.5) / N and (j + 0.5) / N.
  • To determine whether a subsample is inside the triangle, I use a helper lambda called triangle_check, which computes the 2D cross product sign for a triangle edge. I evaluate this for all three edges of the triangle at the subsample’s position.
  • I then check whether the edge signs are consistent. If the signs are not mixed (meaning the point lies on the same side of all three edges), the subsample is inside the triangle. If it is inside, I compute the correct index into the sample_buffer using the pixel’s base index plus the subsample offset (j * N + i), and I store the triangle’s color there.
  • After all triangles are rasterized, resolve_to_framebuffer averages all the subsamples for each pixel and writes the final averaged color to the screen.

Why Supersampling is Useful

  • Supersampling is useful because it reduces aliasing, especially along diagonal and thin edges.
  • At sample rate 1, the rasterizer only tests the center of each pixel. This produces a binary decision: a pixel is either fully inside the triangle or fully outside. That causes jagged, stair-stepped edges.
  • With supersampling, each pixel is divided into multiple subsamples. Some subsamples may lie inside the triangle while others lie outside. When these are averaged together, the final pixel color reflects how much of the pixel is covered. This produces smoother edges and a more natural-looking result.

Modifications to the Rasterization Pipeline

To support supersampling, I made several changes:

  • I introduced the sample_buffer, which stores multiple subsamples per pixel instead of one color per pixel.
  • I updated rasterize_triangle so that it loops over all subsamples within each pixel rather than testing only the pixel center.
  • I ensured that subsamples are written individually into the sample_buffer.
  • I relied on resolve_to_framebuffer to average subsamples into a final color per pixel.
  • For points and lines, I modified fill_pixel so that it fills all subsamples of a pixel with the same color, instead of performing subsample edge testing.

These changes shift the pipeline from per-pixel coloring to per-subsample coloring followed by averaging.

How I Used Supersampling to Antialias My Triangles

  • For each triangle, I subdivide every pixel in its bounding box into an N × N grid of subsamples. I test each subsample using the cross-product edge function to determine whether it lies inside the triangle. Only the subsamples inside the triangle are colored.
  • When resolve_to_framebuffer averages the subsamples, partially covered pixels produce blended colors instead of harsh edges. This smooths the boundary of the triangle.
  • As the sample rate increases, more subsamples are used per pixel, which improves the accuracy of partial coverage and further reduces aliasing artifacts.

Task 4: Barycentric Coordinates

  • Barycentric coordinates are basically a way to describe any point inside a triangle using the triangle’s three vertices. Instead of using normal x and y coordinates, you describe the point as a weighted combination of the three corners. Each vertex gets a weight, usually called alpha, beta, and gamma, and all three weights add up to 1.
  • You can think of it like this: if a point is really close to one vertex, that vertex’s weight will be large, and the other two weights will be smaller. If the point is exactly at a vertex, that vertex’s weight is 1 and the other two are 0. If the point is somewhere in the middle, the weights are more balanced.
  • This is super useful in computer graphics because we can use those weights to smoothly interpolate values across the triangle. For example, if each vertex has a different color (like red, green, and blue), the barycentric coordinates let us blend those colors smoothly across the triangle. Points closer to the red vertex will look more red, closer to green will look more green, and so on. The result is a smooth gradient across the surface of the triangle instead of harsh color boundaries.

Task 5: Pixel Sampling for Texture Mapping

Explain Pixel Sampling and How It Is Used for Texture Mapping

What Pixel Sampling Is

Pixel sampling is how we decide what color to assign to a screen pixel when mapping a texture image onto a triangle. For each pixel inside the triangle, we compute a corresponding texture coordinate (u, v) using barycentric interpolation of the triangle’s vertex texture coordinates.

The (u, v) values are in the range [0, 1], but they usually contain decimal values. Since textures are stored as discrete texels (integer grid positions), we can’t directly index into the image with decimal coordinates. Instead, we need a sampling method to determine what color to return from the texture.

In my implementation, I compute the interpolated (u, v) for each pixel and pass it into tex.sample(). For Task 5, I always sample from mipmap level 0 (the full-resolution texture). Inside the texture class, I implemented two pixel sampling methods: nearest and bilinear.

Nearest-Neighbor Sampling

  • Nearest-neighbor sampling is the simpler method.
  • In my sample_nearest function, I first retrieve the correct mipmap level image. Then I convert (u, v) from normalized coordinates into texture space by multiplying u by the texture width and v by the texture height.
  • After converting into texture space, I round to the nearest integer texel by adding 0.5 before casting to an integer. This effectively selects the texel closest to the (u, v) coordinate.
  • I also clamp the resulting texture coordinates so they stay within valid image bounds. Finally, I return the color of that texel using get_texel.
  • Nearest sampling is fast and straightforward, but because it picks just one texel without blending, it can look blocky or pixelated when the texture is magnified.

Bilinear Sampling

  • Bilinear sampling produces smoother results by blending nearby texels.
  • In my sample_bilinear function, I again start by retrieving the mipmap level image. I convert (u, v) into texture space, but this time I subtract 0.5 so that interpolation happens between texel centers.
  • Next, I compute the base texel coordinates using floor to find the top-left texel of the surrounding square. I then compute the neighboring texels to the right and bottom. All coordinates are clamped to stay within valid bounds.
  • I compute the fractional offsets inside the texel cell, which tell me how far the (u, v) coordinate lies between the texels horizontally and vertically.
  • Then I sample the four surrounding texels: top-left, top-right, bottom-left, bottom-right.
  • I first interpolate horizontally between the top two texels and between the bottom two texels. Then I interpolate vertically between those two blended results. The final color is a smoothly blended combination of all four texels.
  • Bilinear sampling requires more computation than nearest-neighbor sampling, but it significantly improves visual quality, especially when textures are zoomed or stretched.

Why Is This Happening?

The difference comes down to how each method handles interpolation and aliasing. Nearest-neighbor is simple and fast, but it doesn’t smooth anything; it just picks one texel. Supersampling helps by averaging multiple samples, but if each sample is still harsh (like nearest), the improvement only goes so far.

Bilinear already smooths things at the texture level by blending nearby texels. So even with just 1 sample per pixel, it looks good. When you combine bilinear with supersampling, you’re smoothing both the texture details and the pixel edges, which is why that combination looks the best.

Comment on the Relative Differences Between Nearest and Bilinear Sampling

The biggest differences between nearest and bilinear sampling show up when the texture is zoomed in, has small details, or has strong contrast edges. In those situations, nearest sampling can look very blocky and harsh because it does not blend any surrounding texels. It simply picks one color per sample. That makes edges look jagged and can create obvious aliasing. This is especially noticeable at 1 sample per pixel, where there is no supersampling to smooth things out.

Bilinear sampling, on the other hand, blends the four closest texels together based on how close the sampling point is to each one. That blending makes color transitions look much softer and more natural. So if you have diagonal lines, small text, or smooth gradients, bilinear looks clearly better. The difference becomes even more obvious when you zoom in. Nearest exaggerates the pixelation, while bilinear still keeps things relatively smooth.

If you increase the sample rate to something like 16 samples per pixel, the difference between nearest and bilinear becomes smaller. Supersampling already helps reduce aliasing by averaging multiple samples inside each pixel. However, bilinear with 16 samples still looks the best overall because it combines texture blending with pixel averaging.

Overall, bilinear is usually the better choice when visual quality matters, especially when textures are scaled up or down significantly.

Task 6: Level Sampling with Mipmaps

Explain Level Sampling and How It Is Implemented for Texture Mapping

Level sampling is how we decide which mipmap level of a texture to use when rendering a triangle. A mipmap is essentially a stack of the same texture stored at different resolutions. If a triangle appears small or far away on the screen, using the full resolution texture can cause aliasing and unnecessary computation. Instead, we can use a lower resolution mipmap level that better matches how large the texture appears on screen.

In my implementation, I first compute a floating-point mip level using get_level(). To do this, I measure how fast the UV coordinates are changing across screen space. Specifically, I compute the UV difference between the current pixel and its neighboring pixels in the x and y directions. Then I convert those UV differences into texel space by multiplying by the width and height of mipmap level 0.

I take the maximum of those values to estimate the pixel’s footprint in texture space. Then I use log2 of that footprint to compute the mip level. I clamp the result so it never goes below zero.

Once I have the floating-point level, I choose how to use it based on the level sampling mode:

  • L_ZERO: Always use mipmap level 0.
  • L_NEAREST: Round the floating level to the closest integer mip level.
  • L_LINEAR: Take the floor and ceil mip levels and compute a weight to blend between them.

After selecting the mip levels, I sample from each level using either nearest or bilinear pixel sampling, depending on the pixel sampling mode. If I’m in linear mip mode and the two mip levels are different, I blend the two sampled colors together. When combined with bilinear pixel sampling, this results in trilinear filtering.

Tradeoffs Between Pixel Sampling, Level Sampling, and Supersampling

Now that I’ve implemented pixel sampling, level sampling, and supersampling, it’s clear that each technique affects speed, memory usage, and image quality differently.

Pixel Sampling

  • Pixel sampling controls how we fetch colors from a single mipmap level.
  • P_NEAREST is faster because it just grabs one texel. However, it can look blocky or pixelated, especially when zoomed in.
  • P_LINEAR blends between four texels, which looks much smoother. It costs slightly more computation, but the quality improvement is noticeable.

In short: nearest is faster, bilinear is smoother.

Level Sampling

  • Level sampling controls which mipmap resolution we use.
  • L_ZERO always uses the full resolution texture. This is simple and fast, but when zoomed out it causes aliasing because high-resolution details get compressed into small screen areas.
  • L_NEAREST selects the closest mipmap level. This reduces aliasing significantly while remaining efficient.
  • L_LINEAR blends between two mipmap levels. This provides the smoothest result when zooming out, but is slower because it samples twice and blends.

Higher-quality mipmapping reduces aliasing when textures shrink, but it adds extra sampling cost.

Supersampling (Number of Samples Per Pixel)

  • Supersampling handles aliasing at triangle edges. Instead of sampling once per pixel, we sample multiple times and average the results.
  • Higher sample rates significantly improve edge smoothness.
  • However, they require more memory because we store more subsamples per pixel.
  • They also slow rendering because the full sampling pipeline runs multiple times per pixel.

Supersampling is very powerful for reducing jagged edges, but it increases both memory usage and computation time.

Overall Summary

  • Pixel sampling affects how smooth the texture looks at a single resolution.
  • Level sampling affects how textures behave when scaled down.
  • Supersampling affects how smooth triangle edges look.

The highest visual quality occurs when using bilinear pixel sampling, linear mip level sampling, and a high sample rate. However, this combination also requires the most computation and memory.
